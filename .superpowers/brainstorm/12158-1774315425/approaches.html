SDLC Pipeline Engine — Architectural Approaches

Three ways to orchestrate multi-role agent pipelines in Tacticl. Click the one that resonates most.

A
Pipeline Orchestrator + Sub-Sparks RECOMMENDED
A new SdlcPipelineOrchestrator manages the pipeline lifecycle. Each role execution creates a child spark (using existing parentSparkId), so every role automatically gets all existing tracking — tokens, cost, duration, status, execution logs. The orchestrator controls sequencing, artifact handoff, and rework loops.
User Spark (SDLC) ──→ SdlcClassifierSkill ──→ SdlcPipelineOrchestrator │ ┌─────────────────────────────────┘ │ ▼ ┌──────────────┐ Creates child spark per role │ PipelineRun │ with parentSparkId = user spark │ (Firestore) │ └──────┬───────┘ │ ┌────────────┼────────────┬──────────────┐ ▼ ▼ ▼ ▼ [PM Spark] [Arch Spark] [Impl Spark] [Review Spark] ──→ ... (child) (child) (child) (child) │ │ │ │ ▼ ▼ ▼ ▼ Role-specific Role-specific Role-specific Role-specific system prompt system prompt system prompt system prompt + knowledge + knowledge + knowledge + knowledge + tools + tools + tools + tools │ │ │ │ ▼ ▼ ▼ ▼ Artifact ─────→ Artifact ──→ Artifact ────→ Artifact (requirements) (design doc) (source code) (review result) │ REJECT? ──→ Back to [Impl Spark] (new child) ACCEPT? ──→ Continue to [Tester Spark]

New Entities

PipelineRun (collection: pipeline_runs)
sparkId, userId, pipelineType (FULL_SDLC | FOCUSED | SIMPLE),
activatedRoles[], currentRole, roleResults{} (role → {sparkId, status, artifacts}),
reworkCount, reworkOrigins[], status, totalTokens, estimatedCost, startedAt, completedAt
SdlcRoleConfig (embedded in each role skill)
roleName, systemPrompt, knowledgeBase[], fewShotExamples[], availableTools[],
successCriteria, acceptanceCheck, preferredEngine, preferredModel, maxRounds

Strengths

  • Leverages existing Spark tracking (tokens, cost, duration, logs) — no new metrics infra
  • Each role's work is a discrete, auditable unit
  • Dashboard gets per-role data for free via child spark queries
  • Rework loops create new child sparks — visible in the data
  • Pipeline state survives server restarts (Firestore-backed)
  • Parallel role execution possible (e.g., Tester + Security in parallel)

Considerations

  • More Firestore writes (1 spark per role + pipeline doc)
  • Orchestrator is new infrastructure to build and maintain
  • Artifact passing between roles needs a clean contract
B
Single Service with Role Phases SIMPLER
Extend VoiceAgentService with a pipeline mode. When the classifier says SDLC, the service iterates through roles within a single spark — no child sparks. Each role's work is logged as ExecutionLog entries tagged with the role name. Lighter infrastructure, but less granular tracking.
User Spark (SDLC) ──→ VoiceAgentService (pipeline mode) │ ┌───────┘ │ ▼ Single Spark (stays one spark) │ Phase 1: PM Role ──→ ExecutionLog(role=PM, ...) │ Phase 2: Architect ──→ ExecutionLog(role=ARCHITECT, ...) │ Phase 3: Implement ──→ ExecutionLog(role=IMPLEMENTER, ...) │ Phase 4: Review ──→ ExecutionLog(role=REVIEWER, ...) │ │ │ REJECT ──→ Back to Phase 3 (new log entry) │ ACCEPT ──→ Continue │ Phase 5: Test ──→ ExecutionLog(role=TESTER, ...) │ ... until complete

Strengths

  • Minimal new infrastructure — extends existing service
  • Single spark entity per pipeline run (simpler queries)
  • Faster to build

Considerations

  • ExecutionLog wasn't designed for role-level tracking — would need schema changes
  • Harder to get per-role token/cost breakdowns
  • Rework tracking is implicit (log entries), not structural
  • Can't parallelize roles easily
  • Long-running single spark — if it fails mid-pipeline, harder to resume
  • VoiceAgentService gets bloated with pipeline logic
C
Meta-Agent (Agent-of-Agents) FLEXIBLE
A meta-agent using AgenticApiAiEngine whose tools are the roles themselves. The meta-agent receives the user's request and decides which role to invoke, in what order, and whether to loop. Each role is exposed as a tool (invoke_architect, invoke_implementer, etc.). Most flexible but least predictable.
User Spark (SDLC) ──→ Meta-Agent (AgenticApiAiEngine) │ Tools available: ├── invoke_pm(requirements) ├── invoke_researcher(context) ├── invoke_architect(design) ├── invoke_designer(ux) ├── invoke_planner(tasks) ├── invoke_implementer(code) ├── invoke_reviewer(review) ├── invoke_tester(test) ├── invoke_security(audit) ├── invoke_docs(document) ├── invoke_devops(deploy) └── invoke_retro(analyze) │ Meta-agent decides: "This needs PM first, then Architect..." "Reviewer rejected, calling Implementer again..." "Backend only, skipping Designer..."

Strengths

  • Most adaptive — meta-agent reasons about which roles to invoke
  • Handles edge cases naturally (skip roles, reorder, invoke twice)
  • Minimal orchestration code — the LLM IS the orchestrator

Considerations

  • Least predictable — meta-agent might skip critical roles
  • Hardest to track pipeline progress (no fixed stages)
  • Meta-agent token overhead (needs context of entire pipeline)
  • Rework tracking is opaque — meta-agent's reasoning isn't structured data
  • Can't easily build a pipeline funnel dashboard if the pipeline isn't fixed
  • Expensive: meta-agent's context grows with each role's output

My Recommendation: A (Pipeline Orchestrator + Sub-Sparks)

Why A over B: You explicitly need per-role proficiency tracking, rework origin analysis, and cost breakdowns. Child sparks give you all of this through the existing data model. B would require retrofitting ExecutionLog to carry role-level data it wasn't designed for.

Why A over C: The dashboard needs a predictable pipeline with measurable stages. A meta-agent's flexible routing makes it nearly impossible to build pipeline funnel charts, compare role proficiency across runs, or identify systematic rework patterns. You can't optimize what you can't measure.

Hybrid possibility: Start with A's structured pipeline, but use C's meta-agent concept for the classifier only — let the LLM reason about which roles to activate. The pipeline execution itself stays structured and measurable.
