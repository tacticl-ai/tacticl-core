Async Execution & Full Pipeline Traceability

Pipeline runs asynchronously with real-time visibility into every state change, artifact, and decision

1. Async Execution Model

Request → Immediate Response → Background Pipeline

The API returns immediately with the sparkId and pipelineRunId. The pipeline runs on a background thread. The client polls or subscribes for progress.

POST /api/agent/command { text: "Build a notification system with WebSocket...", sessionId: "..." } Response (immediate, ~200ms): { "sparkId": "spark-abc123", "pipelineRunId": "pipeline-xyz789", ← NEW: only present for SDLC sparks "pipelineTier": "FULL_SDLC", "activatedRoles": ["PM", "RESEARCHER", "ARCHITECT", "PLANNER", "IMPLEMENTER", "REVIEWER", "TESTER", "SECURITY", "DEVOPS", "RETRO"], "responseText": "Starting full SDLC pipeline for your notification system. I'll work through 10 stages — you can track progress in real-time.", "status": "PIPELINE_STARTED" } // Pipeline executes asynchronously in background // Client subscribes to updates via WebSocket / FCM / polling

Execution Infrastructure

Pipeline runs on Spring's @Async with a dedicated thread pool. Each role executes sequentially (or in parallel for independent roles like Tester + Security). Pipeline state is Firestore-backed, so it survives server restarts.

@Async("sdlcPipelineExecutor") SdlcPipelineExecutor (ThreadPoolTaskExecutor) PipelineStateManager
SdlcPipelineOrchestrator: @Async("sdlcPipelineExecutor") public void executePipeline(String pipelineRunId) { PipelineRun run = pipelineRunRepository.findById(pipelineRunId); for (SdlcRole role : run.getActivatedRoles()) { // 1. Emit: ROLE_STARTED event emitEvent(run, PipelineEventType.ROLE_STARTED, role); updateState(run, role, RoleStatus.EXECUTING); // 2. Create child spark String childSparkId = sparkService.createChildSpark(run.getSparkId(), role); // 3. Build role context (upstream artifacts + knowledge + examples) RoleContext ctx = buildRoleContext(run, role); // 4. Execute role RoleResult result = roleSkillRegistry.getRole(role).execute(ctx); // 5. Emit: ROLE_COMPLETED event with metrics emitEvent(run, PipelineEventType.ROLE_COMPLETED, role, result.getMetrics()); // 6. Store artifacts artifactService.store(run.getId(), role, result.getArtifacts()); // 7. Evaluate: does downstream role accept this output? if (result.requiresDownstreamValidation() && nextRoleRejects(run, role, result)) { // Rework loop (see section 3) handleRework(run, role, rejectionFeedback); } // 8. Update pipeline state updateState(run, role, RoleStatus.COMPLETED); } // 9. Aggregate metrics, mark complete completePipeline(run); } Thread Pool Config: corePoolSize: 4 // 4 pipelines can run concurrently maxPoolSize: 8 // burst to 8 queueCapacity: 50 // 50 pipelines can queue threadNamePrefix: "sdlc-pipeline-"

Resumability — Crash Recovery

If the server restarts mid-pipeline, the orchestrator picks up where it left off. PipelineRun tracks currentRole and each role's status. On startup, a recovery job finds pipelines in EXECUTING state and resumes from the last incomplete role.

PipelineRecoveryJob (@PostConstruct): 1. Query pipeline_runs WHERE status = EXECUTING 2. For each: a. Find last role with status = COMPLETED b. Resume from next role in sequence c. Emit PIPELINE_RESUMED event d. Continue normal execution loop
PipelineRecoveryJob @PostConstruct startup hook
2. Real-Time Progress — Three Channels
① WebSocket (Existing Infra)

Push pipeline events to connected mobile/web clients in real-time. Same WebSocket infrastructure used for device agent communication.

Events: role_started, role_completed, role_rejected, rework_triggered, checkpoint_requested, artifact_produced, pipeline_completed

Target: Mobile app (live view), Admin console (live dashboard)
② FCM Push (Existing Infra)

Push notifications for key milestones when the user isn't actively watching. Batched to avoid notification spam.

Events: pipeline_started, checkpoint_needed (user action required), pipeline_completed, pipeline_failed

Target: Mobile app (background notifications)
③ Polling API (New Endpoint)

REST endpoint for clients that can't use WebSocket. Returns current pipeline state with full event history.

GET /api/sparks/{id}/pipeline — returns PipelineRun with all events, role statuses, artifacts, and metrics.

Target: Admin console, external integrations, CLI tools

Event Emitter Architecture

All pipeline state changes flow through a single PipelineEventEmitter that fans out to all three channels + writes to Firestore for persistence.

PipelineEventEmitter: emitEvent(PipelineRun run, PipelineEventType type, SdlcRole role, Map metadata): 1. Create PipelineEvent { pipelineRunId, sparkId, userId, eventType, role, timestamp, metadata (tokens, cost, duration, artifacts, errors) } 2. Persist to Firestore → pipeline_events/ collection 3. Fan out: a. WebSocket → push to user's session channel b. FCM → push notification (if milestone event + user not connected) c. In-memory → update PipelineRun.roleResults (denormalized for fast reads) Event Types: PIPELINE_STARTED ROLE_STARTED ROLE_COMPLETED ROLE_REJECTED REWORK_TRIGGERED REWORK_COMPLETED ARTIFACT_PRODUCED CHECKPOINT_REQUESTED CHECKPOINT_RESOLVED PARALLEL_ROLES_STARTED PIPELINE_COMPLETED PIPELINE_FAILED PIPELINE_RESUMED COST_THRESHOLD_WARNING
PipelineEventEmitter PipelineEvent entity WebSocket (device infra) FCM (existing push)

4. Pipeline & Role State Machines

Pipeline States

CREATED → CLASSIFYING → AWAITING_CONFIRMATION → EXECUTING → COMPLETED
Also: EXECUTING → CHECKPOINT → EXECUTING
Failure: EXECUTING → FAILED | CANCELLED

Per-Role States (within a pipeline run)

PENDING → EXECUTING → COMPLETED
EXECUTING → REJECTED → REWORKING → COMPLETED
EXECUTING → FAILED → ESCALATED (user checkpoint)
Skipped: PENDING → SKIPPED (Designer for backend-only)
5. Updated Database Layer — New Collection

pipeline_events/ (NEW — the traceability backbone)

Every pipeline state change is an immutable event. This is the single source of truth for the timeline, the dashboard, and the audit trail. Append-only, never updated.

FieldTypeDescription
idStringAuto-generated event ID
pipelineRunIdStringParent pipeline run
sparkIdStringParent user spark (for cross-referencing)
childSparkIdStringRole's child spark (if applicable)
userIdStringUser who owns the spark
eventTypePipelineEventTypeROLE_STARTED, ROLE_COMPLETED, REWORK_TRIGGERED, etc.
roleSdlcRoleWhich role this event is about
roleIterationint1 = first pass, 2+ = rework iterations
metadataMapFlexible: tokens, cost, duration, artifactId, error, rejectionReason, model, engine
timestampTimestampWhen this event occurred

Why a separate events collection? PipelineRun is a mutable summary (current state). Pipeline events are the immutable audit trail. The dashboard queries events for time-series analytics. The mobile app reads PipelineRun for current status. Both views serve different needs from the same underlying data.
