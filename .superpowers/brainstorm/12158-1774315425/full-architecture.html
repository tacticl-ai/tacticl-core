③ Orchestration Layer — Pipeline Lifecycle (NEW)

SdlcPipelineOrchestrator

The brain of the pipeline. Creates the PipelineRun, iterates through activated roles in order, manages artifact handoff between roles, handles rework loops, and tracks completion.

orchestrate(parentSparkId, classificationResult):
  1. Create PipelineRun in Firestore
  2. For each activatedRole in order:
     a. Create child Spark (parentSparkId = user's spark)
     b. Build role-specific context:
        - Role system prompt + knowledge base + few-shot examples
        - Artifacts from previous roles (requirements, design, code, etc.)
        - Rework feedback (if this is a retry)
     c. Execute role via AiEngineRouterService.executeStep(role.sdlcStep, request)
     d. Evaluate role output against success criteria
     e. Store artifacts in PipelineArtifact
     f. Update PipelineRun.roleResults
  3. If role output is REJECTED by downstream role:
     a. Increment reworkCount
     b. Record rework origin (which role triggered rework)
     c. Jump back to the target role with rejection feedback
     d. Create NEW child spark for the rework iteration
  4. After final role completes:
     a. Run RetroAnalyst role
     b. Mark PipelineRun as COMPLETED
     c. Mark parent Spark as COMPLETED
     d. Aggregate total tokens/cost from all child sparks
SdlcPipelineOrchestrator PipelineArtifactService ReworkTracker

Artifact Flow Between Roles

Each role consumes artifacts from upstream and produces artifacts for downstream. Artifacts are stored in Firestore and referenced by ID — not passed inline (keeps context windows manageable).

PM: requirements + acceptance criteria → Researcher: context + findings → Architect: design doc + API contracts → Planner: task breakdown + execution order → Implementer: source code + commits → Reviewer: review result (ACCEPT/REJECT + comments) → Tester: test results + coverage → Security: security report → Docs: documentation → DevOps: deploy config → Retro: pipeline analysis

Rework Loop Mechanics

When a downstream role rejects (Reviewer rejects code, Tester finds bugs, Security finds vulnerabilities), the orchestrator creates a new child spark for the upstream role with the rejection feedback as context.

Implementer (spark-101) → produces code
Reviewer (spark-102) → REJECTS: "Missing error handling on API calls"
↳ Rework triggered → Implementer (spark-103) with rejection feedback
Implementer (spark-103) → fixes code
Reviewer (spark-104) → ACCEPTS
↳ Continue to Tester

Max rework per role: 3 — after 3 rejections, escalate to user checkpoint for guidance.

④ Role Execution Layer — 12 Specialized Agent Skills (NEW)

SdlcRoleSkill Interface

Each role implements a common interface. The orchestrator calls roles through this contract. Each role has its own system prompt, knowledge base, tools, and success criteria.

interface SdlcRoleSkill {
  SdlcRole getRole();               // PM, ARCHITECT, IMPLEMENTER, etc.
  String getSystemPrompt();           // Role-specific instructions
  List<KnowledgeEntry> getKnowledge();// Domain knowledge base
  List<FewShotExample> getExamples(); // Input→Output examples
  List<String> getAvailableTools();   // Which agent skills this role can use
  String getSdlcStepName();           // Maps to AiSdlcStep for engine routing
  SuccessCriteria getSuccessCriteria();// How to evaluate output quality
  RoleResult execute(RoleContext ctx); // Run the role with upstream artifacts
}

⑤ AI Engine Layer — Execution (Existing + Extended)

Engine Resolution per Role

Each role maps to an SDLC step, which resolves to an engine via the existing AiEngineRouterService. New SDLC steps are added to AiSdlcStep enum and AiSdlcStepDefaults.

AiEngineRouterService AiEngineRegistry AiSdlcStep (add new steps) AiSdlcStepDefaults (add new defaults)
AgenticApiAiEngine (multi-turn tool loop) ApiAiEngineAdapter (single-turn) claude-code-cli engine codex-cli engine

Tool Access per Role

Each role has a filtered tool set. The Implementer gets full device/cloud tools. The PM gets search + browse only. The Retro Analyst gets no tools (pure analysis). Orchestrator passes allowedTools to the engine request.

⑥ Database Layer — Firestore Collections
Collection Status Description Key Fields
sparks/ existing Parent spark (user's request) + child sparks (one per role execution) parentSparkId, type, status, totalTokens, estimatedCost, completedAt
pipeline_runs/ NEW One doc per SDLC pipeline execution. Tracks overall pipeline state, role sequence, rework. sparkId, userId, pipelineTier, activatedRoles[], currentRole, roleResults{}, reworkCount, reworkOrigins[], status, totalTokens, totalCost, startedAt, completedAt, classifierOutput{}
pipeline_artifacts/ NEW Artifacts produced by each role. Referenced by pipeline run + role. Consumed by downstream roles. pipelineRunId, role, sparkId, artifactType (REQUIREMENTS | DESIGN | CODE | REVIEW | TEST_RESULTS | SECURITY_REPORT | DOCS), content{}, version, createdAt
sdlc_role_knowledge/ NEW Knowledge base entries for each role. RAG-indexed. Grows via retro learnings. role, category (BEST_PRACTICE | ANTI_PATTERN | EXAMPLE | RETRO_LEARNING), content, embedding[], source, createdAt
pipeline_rework_events/ NEW Every rework event. Who rejected, why, which role was sent back to. Critical for the dashboard. pipelineRunId, rejectingRole, targetRole, reason, rejectionSparkId, reworkSparkId, reworkIteration, createdAt
execution_logs/ existing Individual tool calls within each role's execution (unchanged) sparkId, toolName, toolInput, toolOutput, tokenUsage, durationMs
agent_audit_log/ modified Add pipelineRunId + role fields to link audit entries to pipeline context + pipelineRunId, + sdlcRole, (existing: userId, commandText, toolsInvoked, success, tokenCount, executionTimeMs)
checkpoints/ existing User approval checkpoints — reused for SDLC confidence gates + rework escalation sparkId, title, description, options, userDecision, decidedAt
