SDLC Pipeline — Agent Roles & Phases

When a spark requires full feature/product development, it flows through these roles. Each role is a distinct agent with its own model, tools, and success criteria. Click to select roles that feel right, or note what's missing.

Phase 1 — Discovery & Definition
1
Product Manager
Translates the user's request into structured requirements. Defines scope, acceptance criteria, success metrics, and constraints. Identifies what "done" looks like.
Outputs: Requirements docAcceptance criteriaScope boundary
↓
2
Researcher
Investigates existing codebase, APIs, libraries, and prior art. Gathers context the downstream roles need. Web search, code analysis, dependency review.
Outputs: Research findingsCodebase contextAPI referencesRisk flags
Phase 2 — Design & Architecture
3
Architect
Designs the technical solution. Component breakdown, data models, API contracts, integration points, technology choices. Makes the hard structural decisions.
Outputs: Architecture designData modelsAPI contractsComponent diagram
↓
4
Designer
UI/UX design when the feature has a user-facing component. Wireframes, user flows, component design. Skipped for backend-only work.
Outputs: WireframesUser flowsComponent specs
↓
5
Planner
Breaks the architecture into ordered implementation tasks. Identifies dependencies, parallelizable work, and critical path. Creates the execution plan.
Outputs: Task breakdownDependency graphExecution orderEffort estimates
Phase 3 — Implementation
6
Implementer
Writes the code. New features, refactoring, bug fixes, integrations. Follows the plan from step 5 and the architecture from step 3. The workhorse of the pipeline.
Outputs: Source codeDatabase migrationsConfig changesCommit history
↓
7
Reviewer
Code review for quality, correctness, patterns, and adherence to architecture. Can request changes (triggers rework back to Implementer). Gatekeeps what gets merged.
Outputs: Review commentsApproval/rejectionRework requests
Phase 4 — Quality Assurance
8
Tester
Writes and runs tests — unit, integration, E2E. Validates against acceptance criteria from step 1. Can reject back to Implementer if tests fail.
Outputs: Test suitesTest resultsCoverage reportsBug reports
↓
9
Security Analyst
Security review and penetration testing. OWASP checks, dependency vulnerability scanning, auth/authz validation, injection testing. Can block deployment.
Outputs: Security reportVulnerability findingsRemediation items
Phase 5 — Delivery
10
Technical Writer
API documentation, README updates, architecture docs, changelog entries, PR descriptions, commit messages. Makes the work understandable to others.
Outputs: API docsREADMEChangelogPR description
↓
11
DevOps Engineer
CI/CD pipeline, deployment configs, infrastructure changes, monitoring setup. Gets the code from repo to production safely.
Outputs: CI/CD configDeploy scriptsInfra changesMonitoring alerts
Phase 6 — Retrospective
12
Retrospective Analyst
Analyzes the full pipeline run. What went well, where rework happened, which roles caused downstream issues. Feeds learnings back into future pipeline runs.
Outputs: Retro reportRework analysisImprovement itemsRole proficiency scores
