Product Requirements Document

Tacticl PDLC v2

Date: 2026-04-11 Status: Draft Author: Gabriel Jimenez

1. Executive Summary

Tacticl v1 ships a Product Development Lifecycle (PDLC) pipeline that runs 12 specialized AI agents inside a single Spring Boot JVM on Cloud Run. It works, but it has architectural ceilings that prevent it from reaching the quality bar required for production software delivery: shared thread pools, hand-coded orchestration, no workspace isolation, no cross-run memory, limited rework iterations, and tight coupling between pipeline logic and LLM provider code.

Tacticl PDLC v2 rebuilds the pipeline on top of cidadel-ai-arbiter, Tacticl's new agent execution factory: each role runs in its own Docker container on Hetzner with a pre-assembled workspace, Claude Code CLI as the execution engine, aggressive multi-candidate generation and rework loops, scoped vector search over past successful runs, a 4-layer knowledge system, and between-agent user checkpoints.

Core Thesis

The goal is not speed of delivery or cost minimization. The goal is the highest possible quality of generated code — production-ready, tested, documented, secure, and consistent with the target codebase's conventions. Every design choice in v2 prioritizes quality, even at significant cost and latency overhead.

2. Problem Statement

2.1 What's Broken in v1

Architecture

  • All 12 roles execute in one JVM thread pool (pdlcPipelineExecutor, 4 core / 8 max). Concurrent pipelines compete for CPU, memory, and Anthropic rate limits at the JVM level.
  • Zero workspace isolation — every pipeline shares the same filesystem and process. Two concurrent IMPLEMENTER runs on different repos theoretically could clobber each other. The safety net is "don't run concurrent PDLC for now."
  • Pipeline orchestration is hand-coded Java. Adding a new role, a new playbook, or a new rework policy requires code changes, compilation, and deploy.
  • LLM routing happens through AiEngineRouterService with provider adapters wired via Spring beans. Each provider is a separate Gradle module. The router code is intertwined with cost tracking, rework detection, and event emission.

Knowledge & Memory

  • No cross-run memory. Every pipeline starts from scratch. An IMPLEMENTER that solved a similar problem last week has no way to recall that solution.
  • KnowledgeBaseService exists but uses Firestore vector search (being decommissioned with the Hetzner migration) and is populated only by the Retro Analyst — most roles have no authored knowledge curated for them.
  • Agents waste turns re-discovering codebase conventions every run. The same file imports, naming patterns, and architectural decisions get re-derived over and over.

Execution Engine

  • Claude Code CLI is invoked as a subprocess from Java via client-claude-code. Subprocess spawn adds 5-11 seconds of overhead per invocation. The persistent session mode exists but the architecture fundamentally wants the CLI to live inside a Node.js runtime, not Java.
  • CLI execution happens on the same Cloud Run instance that serves HTTP requests. A long-running CODE_GENERATION session competes with spark classification, dashboard queries, and device WebSocket handling.

Quality Ceiling

  • Rework loops cap at 3 iterations. Complex tasks that need 5-8 iterations are marked failed.
  • IMPLEMENTER generates one candidate. If that candidate has subtle bugs the REVIEWER doesn't catch, they ship.
  • Tests are generated after implementation (not TDD). Tests become rubber-stamps rather than specifications.
  • No mandatory multi-critic consensus. A single REVIEWER can approve flawed code.
  • No REFACTOR pass after tests pass — functional code is considered "done."

Operational

  • Cloud Run Java image is ~400MB and includes Claude Code CLI, Node.js runtime, Playwright, and every LLM client. Cold starts are slow, deploys are slow.
  • Firestore PDLC state means queries are expensive and limited. No aggregations, no efficient cross-pipeline analytics.
  • PipelineRecoveryJob runs on startup but pipeline state lives in Firestore subcollections that are hard to reason about.

2.2 Why a Refactor, Not a Fix

The v1 problems aren't bugs — they're consequences of the architecture. You can't bolt workspace isolation onto an in-JVM pipeline. You can't make Java a good host for Claude Code CLI. You can't make single-candidate generation produce best-of-3 quality by adding a retry.

The arbiter factory pattern — already proven in strategiz v2-generative — solves all of these at the architecture level. Tacticl v2 adopts it wholesale for PDLC work.

3. Vision & Goals

3.1 Vision

Tacticl is a factory for building software. A user describes what they want in natural language. The factory produces production-ready code — designed, implemented, tested, reviewed, documented, and deployed — using 12 specialized AI agents coordinated through a configurable pipeline. The factory improves over time through a weekly learning loop. The user is in control at every checkpoint.

4. Non-Goals

Explicitly not included in v2 scope:

  • Non-PDLC workloads — social posting, chat/spark execution, video generation, research tasks stay in Java tacticl-core for now. They migrate in later phases.
  • Device-side execution — desktop daemon tactics decomposition is a separate execution plane. PDLC v2 is server-side only.
  • Multi-tenant SaaS — v2 targets single-user and small-team use. Multi-tenant isolation beyond per-user workspace namespacing is v3.
  • General-purpose agent framework — v2 is specifically for PDLC. It's not "run any agent on any task."
  • IDE integration — no VSCode extension, no JetBrains plugin. Chat, voice, and GitHub webhook are the only triggers.
  • Real-time collaborative editing — agents produce PRs; humans review async.
  • Cost minimization — quality is the priority. Cost controls exist but not at the expense of output quality.

5. Users & Personas

5.1 Primary: Solo Developer / Founder

  • Builds products alone or on a small team
  • Uses tacticl to offload implementation work while staying in architectural control
  • Reviews every PR before merging
  • Cares about code quality, maintainability, test coverage
  • Willing to wait minutes (or hours) for a high-quality result
  • Has 1-10 repos connected to tacticl

5.2 Secondary: Product Manager

  • Files feature requests as GitHub issues
  • Tacticl picks up the issue via webhook and runs PDLC
  • PM reviews the resulting PR without needing to write code themselves
  • Uses checkpoints to steer the pipeline when the AI's direction needs adjustment

5.3 Tertiary: Engineering Lead

  • Sets org-wide quality bars via configurable checkpoints, rework limits, and required critics
  • Reviews weekly retro outputs, approves learnings
  • Monitors pipeline cost/quality metrics across the team

6. Use Cases

UC1: Solo Developer Implements a Feature

Trigger: "Add a password reset flow to the auth service. Send a reset link via SendGrid, verify the token, let the user set a new password."

  1. SparkClassifierService classifies as code spark
  2. PdlcClassifierService selects FULL_PDLC playbook
  3. Tacticl-core submits pipeline to arbiter via gRPC
  4. Arbiter runs PRODUCT phase: PM writes PRD → RESEARCHER investigates existing code
  5. Checkpoint: User reviews the PRD, approves
  6. Arbiter runs DESIGN phase: ARCHITECT + PLANNER
  7. Checkpoint: User reviews design, approves
  8. Arbiter runs DEVELOPMENT phase: IMPLEMENTER generates 3 candidates → CRITIC selects best
  9. Arbiter runs TEST phase (parallel): TESTER + REVIEWER + SECURITY_ANALYST — all must approve, rework up to 5 iterations
  10. Arbiter runs REFACTOR pass after TEST approves
  11. Arbiter runs DEPLOY phase: TECHNICAL_WRITER + DEVOPS
  12. RETRO_ANALYST runs post-pipeline
  13. Arbiter creates a PR on user's GitHub repo
  14. Push notification: "Your password reset flow PR is ready"

UC2: Bug Fix from GitHub Issue

Trigger: GitHub issue filed with label tacticl-implement: "Login endpoint returns 500 when username contains emoji"

  1. GitHub webhook fires → tacticl-core receives
  2. Tacticl-core creates a spark from the issue
  3. BUG_FIX playbook selected
  4. RESEARCHER reproduces bug → IMPLEMENTER fixes (multi-candidate) → TESTER writes regression test → REVIEWER approves
  5. PR created, linked to original issue
  6. Issue commented with PR link

UC3: User Rejects an Architecture

  1. User feedback at DESIGN checkpoint: "Don't use SendGrid, we use Mailgun. Also tokens should expire in 15 minutes, not 1 hour."
  2. Tacticl-core forwards feedback with status=rework
  3. ARCHITECT re-dispatched with feedback in context/checkpoint-feedback.md
  4. New design produced, pipeline pauses at checkpoint again

UC4: Rework Loop from Failing Tests

  1. TEST phase runs, IMPLEMENTER's code fails 2 of 8 tests
  2. TESTER marks rework needed, writes feedback
  3. Shell re-dispatches IMPLEMENTER with prior implementation + test failures in context
  4. Loop continues until tests pass or rework counter hits 5
  5. If exhausted → REWORK_ESCALATED event → user checkpoint

UC5: Retro Learning Loop

  1. Weekly cron Sunday 2 AM UTC
  2. RETRO scans past week's archived workspaces
  3. Identifies patterns: "47 pipelines had IMPLEMENTER wasting tokens re-discovering Jackson 3"
  4. Proposes learning: "Add to IMPLEMENTER authored knowledge: 'This codebase uses Jackson 3'"
  5. Writes to MongoDB agent_knowledge with status=proposed
  6. Eng lead reviews, approves
  7. Next IMPLEMENTER run includes learning in workspace/knowledge/learned/patterns.md

7. Functional Requirements

7.1 Pipeline Execution

  • FR1.1: System MUST run the 12 PDLC roles as isolated Docker containers on Hetzner, one container per role dispatch.
  • FR1.2: Each container MUST receive a fresh, pre-assembled workspace via bind mount before starting.
  • FR1.3: Containers MUST run Claude Code CLI as the execution engine.
  • FR1.4: Container images MUST be generic (single cidadel-agent image). Agent identity comes from boot.md + CLAUDE.md.
  • FR1.5: Workspaces MUST be isolated — two concurrent pipelines share zero filesystem state.
  • FR1.6: Failed or completed containers MUST be destroyed after result extraction; workspaces archived for 30 days.
  • FR1.7: Pipeline execution MUST be asynchronous — SubmitPipeline returns in ≤2 seconds with a pipeline ID.

7.2 Role Orchestration

  • FR2.1: Pipelines MUST be defined as JSON files in a registry, not hardcoded Java.
  • FR2.2: Each role definition MUST include: agent type, model, max_turns, resource class, boot template path, CLAUDE.md path, knowledge files, pre-population spec.
  • FR2.3: The shell MUST dispatch roles according to the pipeline's dependency graph, supporting sequential and parallel groups.
  • FR2.4: The shell MUST pass results from completed roles via workspace/context/prior-agent-output.md.
  • FR2.5: The shell MUST support both container-handled agents and shell-handled agents (deterministic compute, no LLM).

7.3 Knowledge System (4 Layers)

Key Design Decision

Vector search is used ONLY for past runs. The user's live repo is explored via Claude Code's native Grep/Glob/Read/Bash — fresher, more precise than RAG.

  • FR3.1: Layer 1 (Live Repo) — shell clones user's repo into workspace/context/repo/ before container starts, fresh every run.
  • FR3.2: Layer 2 (Authored Knowledge) — shell populates workspace/knowledge/authored/ with curated markdown from registry, scoped per agent type.
  • FR3.3: Layer 3 (Learned Patterns) — shell queries MongoDB agent_knowledge for approved learnings, renders into workspace/knowledge/learned/patterns.md.
  • FR3.4: Layer 4 (Past Runs Vector Search) — Qdrant MCP server available inside agent containers via find_similar_runs(query, top_k).
  • FR3.5: Retro agent indexes successful pipelines into Qdrant using Voyage-code-3 embeddings.
  • FR3.6: Vector search MUST NOT be used for exploring user's live repo — agents use native tools.

7.4 Quality Multipliers

The Real Quality Engine

These multipliers matter more than any other technical choice. Cost and time are not constraints here.

  • FR4.1: Multi-candidate generation — IMPLEMENTER generates 3 parallel candidate implementations by default. CRITIC sub-agent selects best. Configurable 1-5.
  • FR4.2: Rework loops — Up to 5 iterations per role. Triggered by REVIEWER rejection, TESTER failures, SECURITY_ANALYST vulnerabilities, or self-signaled rework.
  • FR4.3: Multi-critic consensus — TEST phase runs REVIEWER + TESTER + SECURITY_ANALYST. All three must approve before DEPLOY.
  • FR4.4: Test-first enforcement — In FULL_PDLC, TESTER writes failing tests BEFORE IMPLEMENTER writes code. TDD at the pipeline level.
  • FR4.5: REFACTOR pass — After TEST approves, IMPLEMENTER runs dedicated refactor iteration for code quality.
  • FR4.6: Sub-agent self-correction — Container agents may spawn sub-agents for validation.
  • FR4.7: High max_turns — Default 100 per role.
  • FR4.8: Best models — Opus for ARCHITECT/SECURITY/CRITIC/RETRO. Sonnet for IMPLEMENTER/TESTER/REVIEWER. Haiku never used in PDLC v2.

7.5 Checkpoints & User Control

  • FR5.1: Pipeline definitions may mark any role as requiring a checkpoint.
  • FR5.2: Shell pauses pipeline on checkpoint, writes record to MongoDB, emits CHECKPOINT_REQUESTED.
  • FR5.3: Tacticl-core forwards event to mobile/web via WebSocket + push notification.
  • FR5.4: User responds with approve / reject+feedback / cancel.
  • FR5.5: Tacticl-core relays via gRPC ResolveCheckpoint.
  • FR5.6: Arbiter resumes or re-dispatches rejected role with feedback in context.
  • FR5.7: Mandatory checkpoints in FULL_PDLC: after PM, after ARCHITECT, before DEPLOY.

7.6 Triggers

  • FR6.1: Chat — Existing POST /v1/agent/command
  • FR6.2: Voice — Existing push-to-talk flow
  • FR6.3: GitHub webhook — New POST /v1/webhooks/github, triggered by issue with label tacticl-implement
  • FR6.4: Scheduled cron — Recurring sparks via existing agent_reminders collection

7.7 Output Artifacts

  • FR7.1: Default output is a PR on user's GitHub/GitLab repo.
  • FR7.2: Per-user and per-repo configurable: diff-only | branch-push | pull-request (default) | auto-merge.
  • FR7.3: Auto-merge is opt-in per repo, requires all CI checks to pass.
  • FR7.4: PR description includes: original request, playbook, roles, artifacts, cost breakdown, retro observations.

7.8 State & Recovery

  • FR8.1: All pipeline state persisted to MongoDB pipeline_runs on every state transition.
  • FR8.2: In-memory PipelineTracker caches hot state.
  • FR8.3: On shell restart, hydrate from MongoDB, resume interrupted pipelines.
  • FR8.4: Pipeline execution must be idempotent — resuming picks up from last completed role.

7.9 Retro Learning Loop

  • FR9.1: Retro container runs weekly (Sunday 2 AM UTC).
  • FR9.2: Scans past 7 days of archived workspaces.
  • FR9.3: Identifies patterns: failures, token waste, knowledge gaps, template improvements.
  • FR9.4: Writes proposed learnings to MongoDB with status=proposed.
  • FR9.5: Eng lead reviews in console UI.
  • FR9.6: Approved learnings flow into next workspace assembly.
  • FR9.7: Retro also populates Qdrant past-runs index.

7.10 Observability

  • FR10.1: Every role emits OTel spans: role.start, role.complete, role.rework, role.checkpoint.
  • FR10.2: Token usage tracked per role per pipeline in MongoDB.
  • FR10.3: Cost tracked with per-pipeline ceiling ($100 default) and monthly spending limit.
  • FR10.4: Structured JSONL logs in workspace/logs/.

8.3 Scale

  • NFR-S1: 20+ concurrent PDLC pipelines on current Hetzner infrastructure
  • NFR-S2: Up to 15 concurrent containers per pipeline (parallel TEST phase + sub-agents)
  • NFR-S3: Qdrant scales to ~100K past run chunks, sub-100ms p95

8.4 Cost

  • NFR-C1: Default per-pipeline ceiling: $100
  • NFR-C2: Default monthly user budget: $0 (must be explicitly enabled)
  • NFR-C3: Cost transparency per role in pipeline detail view
  • NFR-C4: Cost is NOT a quality constraint — multipliers never disabled for cost

8.5 Reliability

  • NFR-R1: Pipeline recovery on shell restart via MongoDB-backed state
  • NFR-R2: Container failure auto-retry (1 attempt before escalating to rework loop)
  • NFR-R3: Callback retry with exponential backoff (3 attempts)

8.6 Security

  • NFR-Sec1: Per-pipeline bind mounts, no shared filesystem
  • NFR-Sec2: GitHub tokens per-pipeline env var, never logged, rotated per-workspace
  • NFR-Sec3: Containers run as non-root, no privileged mode
  • NFR-Sec4: All secrets from Vault, never hardcoded
  • NFR-Sec5: SECURITY_ANALYST veto is a hard stop — cannot be overridden without explicit user checkpoint

9. Success Metrics

9.1 Launch Criteria (v2 GA)

  • ☐ FULL_PDLC runs end-to-end with all 12 roles without manual intervention
  • ☐ At least 3 sample PRs pass CI on first run without human changes
  • ☐ Checkpoint flow works end-to-end
  • ☐ Multi-candidate generation produces measurably better code than single-candidate
  • ☐ Rework loop triggers correctly on test failures and reviews
  • ☐ Retro agent populates agent_knowledge with ≥5 proposed learnings
  • ☐ Past runs index in Qdrant supports semantic search with ≤100ms p95
  • ☐ Tacticl-core forwards pipeline progress via WebSocket
  • ☐ All 5 Hetzner infrastructure pieces operational

9.3 Anti-Goals

Things to NOT Optimize For
  • Speed-to-PR — Don't celebrate sub-10-minute pipelines if they produce lower-quality code. A 2-hour pipeline that produces merge-ready code is better.
  • Cost per pipeline — Don't reduce multi-candidate, rework, or max_turns to cut costs.
  • Number of pipelines run — Count merged PRs and first-run CI pass rate, not vanity metrics.

11.3 Qdrant Collection: tacticl_past_runs

Self-hosted Qdrant Docker container on Hetzner, registered in platform-net for internal access from arbiter shell and agent containers.

Point structure:

  • Vector: Voyage-code-3 embedding (1024 dimensions)
  • Payload schema:
{
  "product": "tacticl",
  "pipeline_id": "uuid",
  "agent_type": "IMPLEMENTER",
  "playbook": "FULL_PDLC",
  "chunk_type": "role_prompt | role_output | outcome_summary",
  "success_score": 0.92,
  "merged_to_main": true,
  "tags": ["auth", "oauth", "spring-boot"],
  "timestamp": "2026-04-01T12:34:56Z",
  "content": "... chunk text ..."
}

Indexing strategy:

  • Filter on product + agent_type for scoped retrieval
  • Filter on success_score >= 0.8 (only successful runs indexed)
  • Filter on merged_to_main = true (only user-accepted work)

Query pattern: Agent containers call MCP tool find_similar_runs(query_text, agent_type, top_k=5) → returns top-5 semantically similar chunks from past successful PDLC runs.

Writer: Retro agent weekly job. Reads archived workspaces, chunks content, embeds via Voyage-code-3, upserts to Qdrant.

Reader: Agent containers via local MCP server. The MCP server exposes find_similar_runs as a tool Claude Code can invoke during execution.

11.6 Cross-Product Database Strategy

Since the arbiter is shared between strategiz and tacticl:

Per-product databases (each product owns its pipeline state):

  • strategizDb — strategy generation runs, strategiz-specific collections
  • tacticlDb — PDLC runs, tacticl-specific collections

Rationale: Pipeline execution schemas diverge between products. Don't force a single schema.

Shared cidadelShared database (cross-cutting data):

  • agent_knowledge — retro learnings, scoped by product field at query time
  • pipeline_costs_rollup (future) — cost analytics across products

Rationale: Cross-product visibility into learning patterns; retro scanner can identify cross-product insights ("both strategiz and tacticl waste tokens on the same thing").

11.10 Databases That User Code Needs (NOT Tacticl Infrastructure)

Important Distinction

Sections 11.2–11.9 describe tacticl's own factory infrastructure — storing pipeline state, agent knowledge, operational data. These databases are NEVER accessed by user code. Tacticl does not host production data for user applications.

Case A: User's spark targets an existing repo (majority of PDLC work)

The agent uses whatever database configuration already exists in the repo.

  • Shell clones user's repo into workspace/context/repo/
  • IMPLEMENTER reads docker-compose.yml, application.properties, existing schema
  • Agent writes code that matches the repo's existing database setup
  • TESTER runs the repo's existing test infrastructure (Spring Boot with H2, Node.js with Testcontainers, etc.)
  • Tacticl provides nothing database-related — the repo's setup handles it

Case B: Greenfield spark — building from scratch

The agent picks an appropriate database, generates code + infrastructure, and hands off to the user.

  • ARCHITECT chooses database based on requirements
  • IMPLEMENTER generates application code + docker-compose.yml + migrations + README
  • TESTER uses embedded databases (H2, Fongo, Fake Redis) or Testcontainers for integration tests
  • DEVOPS generates deployment configs (Docker Compose, Kubernetes manifests, etc.)
  • Pipeline produces a runnable PR — user deploys it to their own infrastructure
  • Tacticl does NOT host the running application. Ever.

Testing Strategy for Database-Dependent Code

StrategyWhenHow
Embedded databases (default)Unit tests, simple integration testsH2/HSQLDB for SQL, Fongo for MongoDB, Fake Redis — runs in-memory inside the agent container, zero infrastructure
Testcontainers (escape hatch)Real integration tests where embedded isn't sufficientEphemeral Postgres/MySQL/Mongo/Redis containers spun up inside the agent container's Docker context, destroyed after tests

Container Image Requirements for Testcontainers Support

  • Docker socket mount (v2 default) — /var/run/docker.sock mounted into agent container. Agent spawns sibling containers on host Docker daemon. Simple, privileged.
  • Docker-in-Docker (v2.1 if needed) — nested Docker daemon inside the agent container. Heavier, more isolated.

Security safeguard: Arbiter shell validates docker run commands against a curated allowlist (Postgres, Mongo, Redis, MySQL). Agent containers cannot spawn arbitrary images on the host.

What Is NEVER Done

  • User code never accesses tacticl's MongoDB, Qdrant, or any infrastructure database
  • Tacticl never provisions production databases for user applications
  • Tacticl never hosts user data in its own storage systems
  • In-memory databases (H2, SQLite in-memory, embedded Mongo) are strictly testing tools — never a substitute for real production databases

12. Constraints

  • C1: Hetzner infrastructure only. No GCP, no AWS.
  • C2: Tacticl-core remains Java/Spring Boot as the control plane.
  • C3: cidadel-core is the auth source of truth — don't rebuild in Node.js.
  • C4: Claude Code CLI is the execution engine — not SDK, not direct API.
  • C5: Single Vault endpoint for all secrets.
  • C6: MongoDB + Qdrant only. No Postgres, no Neo4j, no Redis.
